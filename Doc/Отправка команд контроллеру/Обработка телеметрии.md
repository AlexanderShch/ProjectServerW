# Обработка телеметрии

Документ синхронизирован с текущей реализацией обмена в `ProjectServerW`.

## Общая схема

- Все входящие кадры от контроллера сначала попадают в общую очередь ответов (`EnqueueResponse`).
- Разбор кадра (CRC, тип, код, длина) выполняется при ожидании ответа на активную команду.
- В системе действует модель **single in-flight**: одновременно ожидается только одна команда.

## Важные правила

- Сервер отправляет команду и ждёт ответ либо таймаут.
- Пока команда в ожидании, новые команды не конкурируют за чтение ответа.
- Нерелевантные кадры не теряются: временно откладываются и затем возвращаются в очередь.

## Особый случай `SEND_STATE`

Для команды `REQUEST/SEND_STATE (0x03/0x08)` подтверждением считается:

- либо стандартный ответ команды,
- либо кадр телеметрии `TELEMETRY/DATA_OK (0x00/0x01)`.

Это обязательное исключение протокола и должно учитываться в логике ожидания.

## Когда сервер запрашивает состояние/данные

- После установления соединения и установки периода опроса выполняется штатный запрос параметров.
- Автоматический вызов `GET_CMD_INFO` по таймаутам/ошибкам отключен.
- `GET_CMD_INFO` запускается только вручную кнопкой в `DataForm`.

## Диагностика

Если в логе есть сообщения вида «неожиданный тип ответа», это означает, что:

- кадр пришёл, но не соответствовал ожидаемой команде в данный момент;
- кадр должен быть переочередён, а не удалён.
