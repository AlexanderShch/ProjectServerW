# Система команд управления устройством

Документ описывает фактическую систему команд в `ProjectServerW` (по состоянию текущего кода `Commands.h/.cpp` и `DataForm/CommandsDefroster`).

## Формат кадра команды (сервер -> контроллер)

```
[Type][Code][DataLen][Data...][CRC16]
 1 байт 1 байт  1 байт  0..N      2 байта
```

- CRC16 считается по всем байтам кадра, кроме последних 2 байт CRC.
- Формирование кадра: `BuildCommandBuffer(...)`.
- Максимальный размер буфера: `MAX_COMMAND_SIZE = 96`.

## Типы команд

| Тип | Код | Назначение |
|---|---:|---|
| `TELEMETRY` | `0x00` | Подтверждение телеметрии (`DATA_OK`, `DATA_FALSE`) |
| `PROG_CONTROL` | `0x01` | Управление процессом (START/STOP/RESET...) |
| `CONFIGURATION` | `0x02` | Настройка параметров |
| `REQUEST` | `0x03` | Запрос данных/состояния |
| `DEVICE_CONTROL` | `0x04` | Управление устройствами (зарезервировано протоколом) |

## Коды команд

### TELEMETRY (`Type=0x00`)

| Команда | Код | Назначение |
|---|---:|---|
| `DATA_OK` | `0x01` | Телеметрия принята |
| `DATA_FALSE` | `0x02` | Телеметрия отклонена (например CRC) |

### PROG_CONTROL (`Type=0x01`)

| Команда | Код |
|---|---:|
| `START` | `0x01` |
| `STOP` | `0x02` |
| `PAUSE` | `0x03` |
| `RESUME` | `0x04` |
| `RESET` | `0x05` |

### CONFIGURATION (`Type=0x02`)

| Команда | Код | Комментарий |
|---|---:|---|
| `SET_TEMPERATURE` | `0x01` | Установка температуры |
| `SET_INTERVAL` | `0x02` | Интервал измерений (`uint16`) |
| `SET_MODE` | `0x03` | Режим |
| `SET_DEFROST_PARAM` | `0x04` | Один параметр дефроста |
| `SET_DEFROST_GROUP` | `0x05` | Пакет параметров группы (5/6) |

### REQUEST (`Type=0x03`)

| Команда | Код | Комментарий |
|---|---:|---|
| `GET_VERSION` | `0x02` | Версия прошивки |
| `GET_CMD_INFO` | `0x04` | Аудит последней команды |
| `GET_DEFROST_PARAM` | `0x06` | Один параметр дефроста |
| `GET_DEFROST_GROUP` | `0x07` | Пакет параметров группы |
| `SEND_STATE` | `0x08` | Запрос текущего состояния |
| `GET_ALARM_FLAGS` | `0x09` | Регистры аварий устройств/датчиков |

## Особенности обработки ответов

### Общее правило

- Сервер отправляет команду и ожидает соответствующий ответ/событие до таймаута.
- Таймаут зависит от команды (например, `GET_DEFROST_GROUP` имеет увеличенный таймаут).

### `SEND_STATE` (`REQUEST 0x03 / 0x08`)

- По фактическому протоколу контроллера подтверждение `SEND_STATE` приходит телеметрией (`TELEMETRY/DATA_OK`), а не только стандартным `CommandResponse`.
- Поэтому обработчик ожидания должен принимать:
  - либо ответ `REQUEST 0x03 / 0x08`,
  - либо телеметрию `TELEMETRY 0x00 / DATA_OK 0x01`.

### `GET_CMD_INFO`

- `GET_CMD_INFO` в текущей системе запускается **только вручную** кнопкой `button_CMDINFO` в `DataForm`.
- Автоматический запуск `GET_CMD_INFO` по таймаутам/ошибкам отключен.

## Форматы полезной нагрузки (payload)

### `GET_ALARM_FLAGS`

- Запрос: `DataLen = 0`.
- Ответ: `DataLen = 4`.
  - `data[0..1]` -> `Device_AlarmFlags` (`uint16`, little-endian)
  - `data[2..3]` -> `Sensor_AlarmFlags` (`uint16`, little-endian)

### `GET_DEFROST_GROUP`, `groupId=6`

В `DefrostLogGlobalPayload_t` добавлены:

- `debugDisableTargetTStop` (`uint8`)
- `debugDisableDeviceSwitchCheck` (`uint8`)

Порядок полей в конце структуры:

`fishColdTarget_C` -> `debugDisableTargetTStop` -> `debugDisableDeviceSwitchCheck` -> `sensorUseInDefrost[]`

Семантика:

- `debugDisableTargetTStop`:
  - `0` — автостоп по `fishColdTarget_C` включен
  - `1` — автостоп отключен (отладка)
- `debugDisableDeviceSwitchCheck`:
  - `0` — проверка соответствия DO->DI включена
  - `1` — проверка отключена (отладка)

## Имена битов телеметрии (синхронизировано с контроллером)

Первая группа (DI, `T[SQ-1]`, входы модуля I/O) в `DataForm` должна идти строго как `DI_DFR_REGISTERS_t`:

- `Vent1_Left`, `Vent2_Left`, `Vent1_Right`, `Vent2_Right`
- `Ten1_Left`, `Ten2_Left`, `Ten1_Right`, `Ten2_Right`
- `Vent_Out`, `Air_Open`, `Air_Close`, `Gate_Alarm`
- `Gate_Close`, `Gate_Open`, `But_Start`, `But_Stop`

Вторая группа (DO, `H[SQ-1]`, выходы управления) соответствует `DFR_REGISTERS_t`:

- `_V0`, `_V1`, `_V2`, `_V3`
- `_H0`, `_H1`, `_H2`, `_H3`
- `_Out`, `_Inj`, `_Flp`, `_Opn`, `_Dbl`, `_Cls`, `_Wrk`, `_Alr`

Примечание: для `DO15` используется имя `_Alr` (а не `_Stp`/`Alrm`).

## Статусы выполнения (`CmdStatus`)

| Код | Статус |
|---|---|
| `0x00` | `OK` |
| `0x01` | `CRC_ERROR` |
| `0x02` | `INVALID_TYPE` |
| `0x03` | `INVALID_CODE` |
| `0x04` | `INVALID_LENGTH` |
| `0x05` | `EXECUTION_ERROR` |
| `0x06` | `TIMEOUT` |
| `0xFF` | `UNKNOWN_ERROR` |

## Где смотреть код

- `Commands.h` — типы, коды, структуры, фабрики команд.
- `Commands.cpp` — CRC, сборка кадра, разбор ответа.
- `CommandsDefroster.cpp` — отправка команд и ожидание ответов.
- `PacketQueueProcessor.cpp` — обработка входящих кадров и ACK телеметрии.
