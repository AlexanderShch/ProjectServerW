## Старт сервера и передача стартовых команд в полудуплексном режиме

Этот документ описывает, что происходит при запуске сервера ProjectServerW, когда контроллер уже работает, и как в этот момент отправляются стартовые команды в условиях эмуляции полудуплексного обмена (RS‑485 через мост TCP↔UART).

Документ дополняет `ПОЛУДУПЛЕКС_РЕЖИМ.md`, фокусируясь именно на **последовательности событий при старте** и участвующих модулях.

---

## Участвующие модули

### 1. Сервер ProjectServerW (ПК)

- **`SServer.cpp`**
  - Поток приёма для каждого TCP‑подключения:
    - настраивает сокет (порт 8867);
    - в бесконечном цикле:
      - берёт `recvGate` (полудуплексный замок приёма);
      - вызывает `recv()` и накапливает данные в `accumulatedBuffer`;
      - выделяет кадры формата `AA 55 [Type][Len][Payload][CRC16]`;
      - для:
        - **Type=0x00 (телеметрия)** → `PacketQueueProcessor::EnqueueTelemetry(...)`;
        - **Type=0x01 (лог параметров)** → `PacketQueueProcessor::EnqueueControlLog(...)`;
        - **Type=0x01..0x04 (ответы на команды)** → `DataForm::EnqueueResponse(...)`;
      - освобождает `recvGate` в `finally`;
      - делает паузу **300 мс** перед следующей итерацией цикла, чтобы поток UI успевал захватить `recvGate` и отправить стартовые команды.

- **`PacketQueueProcessor.cpp`**
  - Хранит очереди для телеметрии и лога параметров.
  - При получении кадра:
    - **телеметрия**:
      - передаётся в форму `DataForm` (через Invoke/BeginInvoke) для обновления таблицы;
      - обновляются поля `lastTelemetryTime`, `lastTelemetryDeviceSeconds` и флаги наличия данных;
    - **лог параметров (Type 0x01)**:
      - через `EnqueueControlLog` попадает в `DataForm::AppendControlLogToDataRow(...)`, где лог дописывается в строку телеметрии.
    - **Определение автоматического режима**: по биту **_Wrk** (бит 14 выходов DO в телеметрии, поле `H[SQ-1]`): при `_Wrk == 1` — контроллер в автоматическом режиме, кнопки ПУСК/СТОП в состоянии «программа запущена»; при `_Wrk == 0` — «можно запустить». Таймер при отсутствии телеметрии с _Wrk=1 дольше (интервал измерений + 1 с) сбрасывает флаг и переводит кнопки в «можно запустить».

- **`DataForm.cpp`**
  - Основная форма, привязанная к IP‑адресу контроллера:
    - **`OnReconnectSendStartupCommands()`**
      - вызывается из потока приёма через `BeginInvoke`, когда обнаружено переподключение (Reconnect);
      - проверяет, что форма и сокет валидны и стартовые команды ещё не отправлялись на этом сокете (`lastStartupCommandsSocket`);
      - делает паузу **1,5 с** (время на переключение моста TCP↔UART на новый сокет);
      - отправляет стартовые команды:
        - **`SendVersionRequest()`** — команда **GET_VERSION** (REQUEST, Code=0x02);
        - **`SendSetIntervalCommand(intervalSeconds)`** — команда **SET_INTERVAL** (CONFIGURATION, Code=0x02).
      - при успешном выполнении обеих команд фиксирует текущий сокет в `lastStartupCommandsSocket`.
    - **`SendVersionRequest()`**
      - формирует команду REQUEST/GET_VERSION;
      - вызывает `SendCommandAndWaitResponse(...)`;
      - по ответу:
        - извлекает строку версии прошивки;
        - обновляет текст `label_Version` (через `UpdateVersionLabelInternal()` в UI‑потоке);
        - пишет информационное сообщение в лог.
    - **`SendSetIntervalCommand(int intervalSeconds)`**
      - формирует команду CONFIGURATION/SET_INTERVAL (payload: `uint16_t` интервал);
      - вызывает `SendCommandAndWaitResponse(...)`;
      - по успешному ответу:
        - обновляет надпись о текущем интервале;
        - логирует `[OK] Интервал измерений: N с`.
    - **`SendCommandAndWaitResponse(...)`**
      - универсальный метод:
        - вызывает `SendCommand(...)`, который:
          - берёт `recvGate`,
          - делает паузу **150 мс** (задержка разворота линии RS‑485),
          - берёт `sendGate`,
          - выполняет `send()` готового кадра с командой и CRC;
        - затем ждёт ответ в цикле:
          - вызывает `ReceiveResponse(...)` с таймаутом;
          - пропускает «чужие» ответы, при необходимости обрабатывает запоздалые ответы на GET_VERSION / SET_INTERVAL (через `ApplyLateStartupResponse(...)`);
          - по получении подходящего ответа вызывает `ProcessResponse(...)`.

### 2. Контроллер Defrost (STM32)

- **`CommandReceiver.cpp`**
  - Приём и разбор команд с UART4 (через мост TCP↔UART):
    - **`CommandReceiver_ReceiveCommand(...)` / `CommandReceiver_ProcessReceivedData(...)`**:
      - читают кадр: `[Type][Len][Code][Status/DataLen/Data][CRC16]` (контроллер «видит» уже снятый с AA 55 кадр);
      - проверяют длину, CRC, тип и код;
      - вызывают **`CommandReceiver_ProcessCommand(...)`**.
    - **`CommandReceiver_ProcessCommand(...)`**:
      - по Type/Code направляет команду в соответствующий обработчик:
        - **REQUEST / GET_VERSION** → возвращает строку версии `FW_VERSION_STRING`;
        - **CONFIGURATION / SET_INTERVAL** → обновляет интервал передачи телеметрии/лога и возвращает статус `CMD_STATUS_OK`;
        - другие команды (в т.ч. PROG_CONTROL/START/STOP) обрабатываются отдельно;
      - для REQUEST, возвращающих данные (GET_VERSION и др.), формирует ответ сразу внутри обработчика (через `CommandReceiver_SendResponse(...)`).

- **`DefrostControl.cpp` / `Data.cpp`**
  - Управление алгоритмом дефростации:
    - логика автоматического режима;
    - выдача телеметрии и лога параметров в UART4.

## Условия срабатывания «Reconnect: drop old socket»

Сообщение в логе **`Information: Reconnect: drop old socket 10.25.99.101`** формируется в **`SServer.cpp`** при принятии нового TCP‑подключения. Старый сокет при этом не закрывается вызовом `closesocket`, а только переводится в состояние «полузакрыт» через **`shutdown(prevSocket, SD_BOTH)`**, чтобы старый поток приёма по этому сокету мог корректно завершиться (recv вернёт ошибку/0) и сам вызвал `closesocket`.

### Последовательность проверок

1. **Принято новое входящее соединение** на порт 8867, получены `clientSocket` и IP клиента `clientIPAddress` (например, `10.25.99.101`).

2. **Поиск формы по IP**  
   Вызывается **`DataForm::FindFormByClientIP(clientIPAddress)`**:
   - перебираются все формы в `formData_Map`;
   - возвращается **GUID первой формы**, у которой `ClientIP != nullptr` и `ClientIP->Equals(clientIPAddress)`.
   - Если такой формы нет, возвращается пустая строка.

3. **Определение сценария**
   - **`existingGuid` не пустой** → считается **reconnect**: устройство с тем же IP подключается снова (новый TCP‑сокет), форма переиспользуется.
   - **`existingGuid` пустой** → новое подключение: создаётся новая форма, новый GUID.

4. **Получение формы по GUID**  
   **`DataForm::GetFormByGuid(guid)`** возвращает указатель на форму `df`.

5. **Условия для «drop old socket»**  
   Сообщение **«Reconnect: drop old socket {IP}»** выводится **только если** одновременно выполнены:
   - **`df != nullptr`** и форма не в состоянии Disposed/Disposing;
   - **`prevSocket = df->ClientSocket`** — у формы запомнен предыдущий сокет;
   - **`prevSocket != INVALID_SOCKET`** — этот сокет был реально назначен (было хотя бы одно успешное «привязывание» сокета к форме);
   - **`prevSocket != clientSocket`** — новое подключение пришло по **другому** сокету (новый дескриптор).

Тогда выполняется **`shutdown(prevSocket, SD_BOTH)`** и пишется лог **«Reconnect: drop old socket {clientIPAddress}»**. После этого форме назначаются новый сокет и IP: **`df->ClientSocket = clientSocket`**, **`df->ClientIP = clientIPAddress`**.

### Когда это бывает

- **Первый запуск сервера при уже работающем мосте (EW11A и т.п.):**  
  Первое подключение с IP `10.25.99.101` создаёт форму и записывает в неё `ClientIP` и `ClientSocket`. Если мост или клиент разрывает соединение и сразу устанавливает новое с тем же IP (например, перезапуск сервера, переподключение моста), при приёме **второго** соединения `FindFormByClientIP` находит ту же форму по IP, а `df->ClientSocket` ещё указывает на **старый** сокет. Тогда срабатывает «drop old socket» и форма переключается на новый сокет.

- **Важно:** чтобы `FindFormByClientIP` что‑то нашёл, у какой‑то формы уже должен быть установлен **`ClientIP`**, совпадающий с IP нового подключения. `ClientIP` выставляется в этом же обработчике (при первом и при повторном подключении), поэтому сценарий «drop old socket» возможен только когда **уже было** предыдущее подключение с этого IP, форма для него создана и ей был присвоен этот IP (и сокет).

### Итог

| Условие | Назначение |
|--------|------------|
| Есть форма с `ClientIP == IP` нового подключения | Reconnect: переиспользуем форму. |
| У этой формы `ClientSocket != INVALID_SOCKET` и `ClientSocket != clientSocket` | Новое соединение по другому сокету → «drop old socket»: закрываем старый сокет через `shutdown`, логируем, привязываем форму к новому сокету. |
| После этого `isReconnect && df->IsHandleCreated` | В UI‑потоке формы планируется **`OnReconnectSendStartupCommands()`** (GET_VERSION, SET_INTERVAL и т.д.). |

## Цикл передачи контроллера и окно для команд дефростера

По временной диаграмме обмена контроллер ↔ сервер (RS‑485 через мост TCP↔UART) цикл контроллера выглядит так:

1. **Контроллер передаёт телеметрию** — примерно **27,5 мс**.
2. **Контроллер переходит в приём** и ждёт подтверждение (DATA_OK/DATA_FALSE) от сервера.
3. **Ответ сервера** обычно приходит в интервале **50–60 мс** после окончания телеметрии.
4. **Через 4–5 мс после приёма ответа** контроллер передаёт **блок лога параметров** — примерно **38 мс**.
5. **Цикл повторяется каждую секунду** (интервал задаётся SET_INTERVAL).
6. **Если ответа от сервера нет в течение 100 мс**, контроллер всё равно отправляет лог параметров примерно через **104 мс** после окончания телеметрии.

Таким образом, в каждом цикле есть два участка, когда линия свободна для передачи **от сервера к контроллеру** (контроллер в режиме приёма):

| Окно | Когда | Длительность | Назначение |
|------|--------|--------------|------------|
| **Окно 1** | Сразу после окончания приёма телеметрии на сервере, до начала передачи лога контроллером | примерно **0–60 мс** после конца телеметрии (если сервер отвечает быстро — до ~60 мс; контроллер начнёт лог через 50–60 мс + 4–5 мс после своего приёма ACK) | Отправка **DATA_OK/DATA_FALSE** по телеметрии; при необходимости — **команды дефростера** (GET_VERSION, SET_INTERVAL, START, STOP и т.д.), чтобы они успели дойти до контроллера до начала передачи лога. |
| **Окно 2** | После окончания приёма лога параметров на сервере до следующей телеметрии | примерно **≈ 900 мс** (оставшаяся часть секундного интервала) | Самое широкое окно: контроллер не передаёт, сервер может отправлять команды без риска столкнуться с телеметрией или логом. |

**Предложение:** серверу имеет смысл **синхронизироваться с циклом передачи дефростера** и выделить для команд дефростера одно или оба окна:

- **Окно 1** — использовать сразу после обработки телеметрии: освободить `recvGate`, по возможности сразу отправить DATA_OK и при наличии очереди команд — отправить команду (или несколько, с учётом времени разворота линии и ограничения ~60 мс). Текущая реализация уже даёт это окно за счёт паузы 300 мс в потоке приёма после разбора буфера и задержки 150 мс перед `send()` в `SendCommand`; при этом команды не должны отправляться во время приёма телеметрии или лога (удерживается `recvGate`).
- **Окно 2** — после приёма и разбора кадра лога параметров снова освобождается `recvGate` и выполняется пауза 300 мс; в этот период сервер может отправить команды, и до следующей телеметрии остаётся почти вся секунда, то есть окно максимально широкое.

Итог: текущая схема с **recvGate** и паузами уже привязывает отправку команд к «тихим» участкам цикла; при необходимости можно формально зафиксировать в коде или конфигурации использование **окна 1** (сразу после телеметрии) и **окна 2** (после лога) и при отправке команд учитывать, что в окне 1 доступно не более ~60 мс до начала передачи лога контроллером.

## Последовательность событий при запуске сервера (контроллер уже в авто‑режиме)

Рассмотрим типичный сценарий:

1. **Контроллер работает в автоматическом режиме** и с некоторым интервалом передаёт:
   - телеметрию (Type 0x00);
   - лог параметров процесса (Type 0x01).

2. **Сервер ProjectServerW запускается**:
   - слушает порт 3487 (управляющее соединение);
   - при первом подключении моста TCP↔UART создаёт поток приёма (порт 8867).

3. **Первый пакет телеметрии/лога**:
   - мост TCP↔UART получает данные от контроллера и отдаёт их серверу по TCP;
   - поток приёма (`SServer.cpp`) принимает байты:
     - выделяет кадр `AA 55 [Type][Len][Payload][CRC16]`;
     - по Type:
       - 0x00 → телеметрия в `DataForm`;
       - 0x01 → лог параметров в `DataForm`;
     - по завершении обработки освобождает `recvGate` и спит **300 мс**.

4. **Создание/пробуждение формы `DataForm`**:
   - после первого пакета вызывается логика открытия/переиспользования формы;
   - форма отображается пользователю;
   - логирует `Information: The DataForm has been opened successfully!`.

5. **Обнаружение переподключения**:
   - поток приёма фиксирует, что IP‑адрес тот же, но сокет изменился;
   - логирует `Information: Reconnect: drop old socket ...`;
   - через `BeginInvoke` вызывает на форме `OnReconnectSendStartupCommands()`.

6. **Пауза для моста TCP↔UART (1,5 с)**:
   - `OnReconnectSendStartupCommands()`:
     - убеждается, что форма и сокет валидны, и стартовые команды ещё не отправлялись на этом сокете;
     - выполняет `Thread::Sleep(1500)`, чтобы дать мосту время переключить внутреннее состояние на новый TCP‑сокет.

7. **Отправка GET_VERSION**:
   - форма вызывает `SendVersionRequest()`:
     - внутри `SendCommandAndWaitResponse()`:
       - поток UI берёт `recvGate`;
       - ждёт 150 мс (разворот полудуплекса);
       - берёт `sendGate`;
       - отправляет команду `REQUEST / GET_VERSION` по TCP;
       - освобождает `sendGate` и `recvGate`;
     - поток приёма на контроллере принимает команду, обрабатывает её и отправляет ответ.

8. **Получение и обработка ответа GET_VERSION**:
   - поток приёма на сервере:
     - выделяет кадр с Type 0x03 (REQUEST), Code 0x02 (GET_VERSION);
     - передаёт его в `DataForm::EnqueueResponse(...)`;
   - `SendCommandAndWaitResponse()` читает ответ через `ReceiveResponse(...)` до истечения таймаута (6 с);
   - по успешному ответу:
     - `ProcessResponse(...)` логирует:
       - `[OK] Команда Type=0x03 (REQUEST), Code=0x02 (GET_VERSION) успешно выполнена`;
       - строку версии прошивки;
     - `SendVersionRequest()` обновляет `label_Version` на форме.

9. **Отправка SET_INTERVAL**:
   - сразу после GET_VERSION форма вызывает `SendSetIntervalCommand(intervalSeconds)`:
     - формируется команда CONFIG/SET_INTERVAL с текущим значением интервала;
     - вызывается `SendCommandAndWaitResponse()` (аналогично GET_VERSION, но с обычным таймаутом 2 с);
   - контроллер обновляет интервал и возвращает подтверждение.

10. **Получение и обработка ответа SET_INTERVAL**:
    - поток приёма на сервере выделяет кадр:
      - Type 0x02 (CONFIGURATION), Code 0x02 (SET_INTERVAL);
    - `SendCommandAndWaitResponse()` получает ответ, `ProcessResponse(...)` логирует:
      - `[OK] Команда Type=0x02 (CONFIGURATION), Code=0x02 (SET_INTERVAL) успешно выполнена`;
      - «Статус: 0x00 (OK) – Команда успешно выполнена».
    - `SendSetIntervalCommand()` обновляет текст `[OK] Интервал измерений: N с`.

11. **Дальнейшая работа**:
    - сервер продолжает принимать телеметрию и лог параметров;
    - **автоматический режим** сервер определяет по биту **_Wrk** (бит 14 выходов DO в телеметрии, поле `H[SQ-1]`): при `_Wrk == 1` — контроллер в автоматическом режиме, кнопки ПУСК/СТОП показывают «программа запущена»; при `_Wrk == 0` — «можно запустить»;
    - по приходу лога (только в авто‑режиме) `AppendControlLogToDataRow(...)` дописывает лог в последнюю строку телеметрии (состояние кнопок задаётся по _Wrk в телеметрии, не по логу);
    - при отсутствии телеметрии с _Wrk=1 дольше (интервал измерений + 1 с) таймер сбрасывает флаг автоматического режима и переводит кнопки в «можно запустить».

## Временная диаграмма (упрощённая)

Ниже текстовая диаграмма с ключевыми событиями при запуске сервера, когда контроллер уже в автоматическом режиме:

```text
Время → 

Контроллер (UART4, RS-485):
    |---- телеметрия ----|------- телеметрия+лог -------|------ ... -----> 

Мост TCP↔UART:
    принимает с UART4                принимает
    и пересылает →  [AA 55 ...]  →   и пересылает → ...

Сервер, поток приёма (SServer.cpp):
    recv() → кадр (телеметрия/лог)
        ↓
    EnqueueTelemetry / EnqueueControlLog
        ↓
    освобождение recvGate
        ↓
    Sleep(300 мс)
        ↓
    (следующий recv)

UI (DataForm):
    получение первой телеметрии → открытие/пробуждение формы
        ↓
    Reconnect: drop old socket
        ↓  (BeginInvoke)
    OnReconnectSendStartupCommands()
        ↓
    Sleep(1500 мс)  ← время на переключение моста TCP↔UART
        ↓
    SendVersionRequest():
        Enter(recvGate)
        Sleep(150 мс)  ← разворот линии (полудуплекс)
        Enter(sendGate)
        send(GET_VERSION)
        Exit(sendGate)
        Exit(recvGate)
        ↓
    ожидание ответа (до 6 с):
        ReceiveResponse() ← ответ GET_VERSION от контроллера
        ProcessResponse() → лог + обновление label_Version
        ↓
    SendSetIntervalCommand()
        (аналогично, с обычным таймаутом)
```

Эта диаграмма показывает:

- где и как учитывается **полудуплекс** (замок `recvGate` + пауза 150 мс перед `send()`);
- где добавлена **пауза 1,5 с** после переподключения, чтобы мост TCP↔UART успел переключиться на новый сокет;
- как стартовые команды (**GET_VERSION**, **SET_INTERVAL**) входят в общий сценарий запуска сервера при уже работающем контроллере.
